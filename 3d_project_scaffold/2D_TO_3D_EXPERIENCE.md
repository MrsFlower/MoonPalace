# 从 2D 互动小说到 3D 自动化场景引擎的经验提炼与踩坑总结

在前期开发的 2D AI驱动视觉小说引擎项目中，我们积累了大量关于大语言模型（LLM）结构化生成、WebAssembly（Moonbit）状态机管理以及前端交互与数据流转的宝贵经验。这些经验将成为接下来构建 **基于生成式AI与RayLib库的3D游戏场景自动化构建系统** 的核心基石。

以下是为 3D 项目提供的关键经验总结与避坑指南：

## 1. 大模型（LLM）结构化生成的“两阶段”策略（极度重要）

在 2D 项目中，我们发现要求大模型一次性输出既包含丰富逻辑（剧情、多分支）又严格符合 JSON 语法的超长文本，往往会导致逻辑断裂（如选项跳跃、节点缺失）或 JSON 括号不闭合。
- **经验提炼：** 在 3D 项目的“AI Compiler”端，必须采用**思维链（CoT）或多阶段流水线**。
- **3D 项目应用：**
  - **阶段一（创意生成）：** 让 LLM 仅输出自然语言的场景描述大纲与拓扑网络（如：“矿镇分为上层缆车区、中层居住区和下层矿洞，三个区域互相连通”）。
  - **阶段二（数据编译）：** 再次请求 LLM，将其严格转换为包含坐标系映射、长宽高参数和剧情触发点的 `world_data.json`。

## 2. 跨语言边界通信（FFI）的数据传递与防截断机制

在 2D 项目的 Moonbit 与 JS 通信中，我们在传递超长 Base64 图片字符串时遭遇了严重的 `Maximum call stack size exceeded` 和 JSON 解析截断错误。
- **经验提炼：** WebAssembly 与宿主环境（在 3D 中将是 Moonbit 与 Raylib 的 C-FFI）交换大型数据（如庞大的 3D 点云/网格数据）时，**绝对不能依赖简单的字符串全量拷贝或粗暴的循环映射**。
- **3D 项目应用：**
  - 对于海量的离散空间数据（顶点、法线、碰撞体），应该通过 **指针共享（Shared Memory）** 或 **分块传递（Chunking）** 的方式。
  - 对于 JSON 数据配置，需要在 Moonbit 侧实现一个健壮的解析器，并在传递前确保特殊字符（如换行符、引号）的绝对安全转义。

## 3. 异步操作与主线程阻塞（卡顿防治）

2D 项目在加载新背景图片或重构 DOM 时，如果未妥善处理异步状态，极易导致点击按钮后 UI 假死（Freeze）。
- **经验提炼：** 资源加载必须彻底异步化，且状态机的切换逻辑必须与渲染层的更新逻辑解耦。
- **3D 项目应用：**
  - 在加载 `.ply` 或生成的网格模型时，需引入多线程或异步加载机制（Raylib 支持）。
  - Moonbit 作为逻辑层，应该只抛出“状态变更事件”，由 C/Raylib 渲染层去平滑过渡（例如：淡入淡出、摄像机缓动），防止一瞬间载入几万个体素导致画面帧率暴跌（FPS Drop）。

## 4. 全局容错与 Debug 体系（Log System）

由于涉及 AI 接口的不稳定性和多语言跨界（Python -> JSON -> Moonbit -> Raylib），一旦某个环节断裂，很难排查。在 2D 项目中，我们通过劫持 Console 并在界面渲染红色 Debug 面板，成功解决了复杂的事件死锁问题。
- **经验提炼：** 必须在项目初期就建立一个全局的、跨语言的日志拦截与追踪体系。
- **3D 项目应用：**
  - **Python端：** 需要对 API 调用失败（如网络、跨域、额度耗尽）做强力捕获，并允许快速回退到 Mock 数据（如生成一个默认的正方体房间）。
  - **Moonbit端：** 防止数组越界导致的 Panic（比如状态机在空历史中 pop），所有可能失败的 FFI 调用必须包裹异常处理。

## 5. 提示词工程（Prompt Engineering）的“物理约束”

2D 项目的经验表明，LLM 的空间想象力极弱，容易生成前后矛盾的视觉描述。
- **经验提炼：** 在引导生成图像或 3D 数据时，Prompt 必须带有极强的**物理规则约束**。
- **3D 项目应用：**
  - 调用 SDXL + ControlNet 时，不仅要描述材质和风格，更要在 Prompt 中明确指定视角（Isometric, Top-down）和深度层次。
  - 在生成 `world_data.json` 时，明确规定坐标系的范围（例如 X/Z 轴在 -100 到 100 之间，Y 轴代表高度），并要求节点之间的距离必须符合逻辑常理。
